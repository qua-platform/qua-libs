Technical briefing

CS_4 QUAM configuration

Two-qubit superconducting stack with a flux-tunable coupler and an additional microwave drive on the coupler, using QUAlibrate quam_config and a targeted extension to the standard quam-builder model.

1 Executive summary

We are configuring the CS_4 OPX1000 cluster for a compact two-qubit experiment: one flux-tunable transmon, one fixed-frequency transmon, and a tunable coupler between them. The coupler must support both flux bias (standard) and a dedicated XY microwave line for new control experiments (for example continuous-wave spectroscopy on the coupler).

CTO takeaway: The approach reuses the existing QUAM / wirer pipeline for most of the wiring and avoids forking quam-builder until upstream supports coupler XY natively. 2 Logical system topology

  • Q1: readout resonator, XY drive, individual flux line.
  • Q2: readout resonator, XY drive; no qubit flux line.
  • Coupler: dedicated flux line plus dedicated XY (microwave) line for advanced protocols.

3 Software stack and workflow

Stack

  • QUAM — machine state, JSON round-trip, generate_config()
  • quam-builder — transmon / pair types, build_quam, CZ helpers
  • qualang_tools.wirer — connectivity allocation
  • quam_config (this repo) — site-specific Quam, generate_quam.py, populate scripts

Workflow

  1. Run generate_quam.py → wiring + skeleton state.json
  2. Post-process coupler → attach composite z + xy
  3. Run populate_quam_*.py → frequencies, pulses, CZ macros
  4. Calibrations consume Quam.load() in nodes

4 Gap vs upstream quam-builder

Today, TunableCoupler is implemented as a single flux channel. There is no first-class “coupler XY” line type in the wirer, and pair align() only registers one coupler element name.

Risk if ignored: Experiments that need coupler.xy.play("cw")-style APIs would be forced into ad-hoc QUA elements outside QUAM, losing serialization, reuse across calibrations, and alignment guarantees.

Mitigation (local extension)

  • Composite TunableCouplerWithXY: holds z: TunableCoupler and xy: XYDriveMW; delegates flux helpers to z.
  • Subclass pair: FluxTunableTransmonPairXYCoupler — align / wait include both channel names.
  • Quam overrides qubit_pair_type so the builder instantiates the extended pair type.
  • generate_quam.py performs a post-build swap of the coupler object and wires MW port O4 for XY.

5 Lab validation — resonator spectroscopy (CS_4)

After closing other quantum machines (00_close_other_qms), we ran the standard QUAlibrate node calibrations/1Q_calibrations/02a_resonator_spectroscopy.py against the live cluster using Quam.load() from this configuration.

What broke (first attempt)

Program generation failed inside initialize_qpu → FluxTunableQuam.set_all_fluxes: for flux_point="joint" the stock code read target.z.joint_offset for every active transmon. Q2 has no qubit flux line (z is absent), so Python raised AttributeError: 'NoneType' object has no attribute 'joint_offset'.

What we changed

  • quam_config/my_quam.py — Quam overrides: set_all_fluxes skips per-qubit z access when the target has no flux; still applies global joint-idle where flux exists, and only calls settle() when z is present.
  • Same file — apply_all_flux_to_zero: guard so active fixed-frequency qubits do not go through q.z.to_zero() (upstream would also fail there).

What succeeded (second attempt)

  • Hardware path: QOP health check passed; QM opened; program queued and completed with a clean execution report.
  • Readout fit — Q1: automated fit reported success (logged resonator ~5.50 GHz, FWHM on the order of a few MHz — values depend on placeholder vs chip-tuned state).
  • Readout fit — Q2: automated fit returned NaN (failure). This points to readout guess / SNR / multiplexing or placeholder frequencies rather than the flux-initialization bug; engineering follow-up is standard spectroscopy tuning, not the coupler-XY extension.

Operational notes

  • Tooling: runtime warned that installed qm-qua is older than the QOP pairing recommends; plan a controlled upgrade when convenient.
  • QUAlibrate DB: saving machine state to the project database was skipped (no database connection configured); local quam_state and run-folder copies still saved.
  • Expected noise: quam-builder emits user warnings when joint-idle skips Q2 because it has no z; harmless once flux helpers are guarded.
CTO takeaway: The CS_4 QUAM model is now exercised on real hardware for a representative 1Q calibration. One real mixed-architecture sharp edge was found and fixed in our thin extension layer; remaining Q2 fit work is normal calibration hygiene.
