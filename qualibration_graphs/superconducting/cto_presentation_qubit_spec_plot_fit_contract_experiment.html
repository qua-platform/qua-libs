03a qubit spectroscopy: plot layer is fused to the fit pipeline

We validated three engineering claims about 03a_qubit_spectroscopy.py and calibration_utils/qubit_spectroscopy/plotting.py: there is no independent “plot raw IQ” path; matplotlib traces read fit.I_rot (output of analysis), not ds.IQ_abs; and the node always hands plotting both ds_raw (post-process_raw_dataset) and ds_fit. A small Python harness reproduced the contracts without touching create_qua_program.

Artifact

calibrations/1Q_calibrations/experiment_plot_fit_iq_contracts.py — static source checks plus three runtime probes (Agg backend).

Verdict

All hypotheses confirmed as stated; one nuance: “raw” in UI text means processed volts / IQ_abs on ds_raw, while the drawn spectrum line is still I_rot from ds_fit.

Run status

Script exit code 0; logs may show QM session init from import side-effects only — no OPX program executed by the harness.

Hypotheses

What we set out to prove

#ClaimOutcome
1 Plotting has no dedicated path for raw demod IQ (e.g. IQ_abs vs frequency) separate from the fit workflow. Confirmed: no ds.IQ_abs / ds.I / ds.Q in trace code; only ds.detuning feeds the Lorentzian x grid.
2 Plotting and fitting are surgically coupled: the visible line uses variables that only exist after fit_raw_data (e.g. I_rot). Confirmed: .plot() calls chain off fit; fit=None crashes before any draw.
3 Trace naming is fixed to the analysis convention (I, Q in analysis; I_rot in plot). Confirmed: plot hard-codes I_rot on the fit dataset; analysis defines I_rot from I and Q.

Node — single call site; no alternate plot entry.

Method

How the harness exercises the contracts

  • Static: inspect.getsource(plot_individual_data_with_fit) — assert no ds.IQ_abs / ds.I_rot substring; assert 03a file text contains the dual-argument plot_raw_data_with_fit(..., ds_fit) pattern.
  • Runtime A: call plot_individual_data_with_fit(..., fit=None) — expect immediate AttributeError on fit.assign_coords (the if fit: guard does not wrap the main .plot calls).
  • Runtime B: minimal xr.Dataset fit with Lorentzian scalars but without I_rot — expect AttributeError (Dataset has no I_rot).
  • Runtime C: ds carries only a detuning coordinate (no I/Q/IQ_abs); fit carries a synthetic I_rot — plot completes, proving the trace does not read IQ from ds.

Command: PYTHONPATH=<superconducting> then python calibrations/1Q_calibrations/experiment_plot_fit_iq_contracts.py.

Nuance for stakeholders

The object named ds_raw at plot time is already processed (volts, IQ_abs, etc.). The experiment still shows that the matplotlib line is not wired to those fields; it is wired to I_rot living on the fit/merged dataset.

Recommendations

If we want looser coupling later

  • Rename APIs for honesty — e.g. plot_rotated_quadrature_with_fit to signal that the “data” trace is post-rotation, not raw counts.
  • Optional raw panel — add an explicit code path: ds["IQ_abs"].plot(...) (or I/Q) behind a parameter, independent of Lorentzian overlay.
  • Guard fit — wrap all fit. dereferences in one guard or require non-None ds_fit before any plotting run action.
  • Regression — keep experiment_plot_fit_iq_contracts.py in CI (fast, no cluster) to catch accidental rewiring.
