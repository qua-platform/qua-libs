Sweep axis rename: why detuning is not “just a label”

In 03a_qubit_spectroscopy.py, the second entry in node.namespace["sweep_axes"] was renamed from "detuning" to "drive_freq_offset_hz". Hardware acquisition and data fetching completed normally. Analysis failed immediately afterward because multiple layers assume the literal dimension name detuning — there is no indirection or parameter today.

Business takeaway

Calibration UX and library code bake in axis names. Renaming for clarity without coordinated refactors yields a late, opaque xarray error, not a compile-time guard.

Technical takeaway

The contract is: sweep_axes keys → xarray dimension names → analysis dim= and ds.detuning accessors. One rename breaks the chain.

Status after doc

Production script was restored to "detuning" after verification; this deck is the record of the experiment.

Single-line surface

What changed in the node

Only the dictionary key for the frequency sweep axis inside create_qua_program was altered. QUA logic (same dfs array, same buffers) was unchanged. 

node.namespace["sweep_axes"] = {
    "qubit": xr.DataArray(qubits.get_names()),
    "drive_freq_offset_hz": xr.DataArray(dfs, attrs={...}),  # was "detuning"
}

What stayed green

  • Program build — independent of axis labels.
  • QOP execution — job finished; execution report reported no errors.
  • XarrayDataFetcher — builds a dataset whose I / Q variables use the new dimension name.
Data path

How the name propagates

sweep_axes dict keys
    → XarrayDataFetcher.initialize_dataset / update_dataset
        → coordinate + dimension names on I, Q (and n)

process_raw_dataset(ds_raw, node)
    → convert_IQ_to_V(ds, qubits)          # still OK (I, Q exist)
    → add_amplitude_and_phase(ds, "detuning", ...)   # HARDCODED dim name

The fetcher is innocent: it faithfully uses whatever string appears in sweep_axes as the non-qubit dimension. The failure is a string contract mismatch between that dict and downstream Python.

Three surfaces

Where detuning is assumed today

Layer Location Examples
Repo analysis qualibration_graphs/superconducting/calibration_utils/qubit_spectroscopy/analysis.py add_amplitude_and_phase(ds, "detuning", …); ds.detuning; .mean(dim="detuning"); peaks_dips(..., dim="detuning"); assign_coords(..., ["qubit","detuning"])
Repo plotting …/qubit_spectroscopy/plotting.py ds.detuning; fit.detuning; derived detuning_MHz coord from fit.detuning
Shared library qualibration_libs/data/processing.py (installed) add_amplitude_and_phase(ds, dim: str) — caller passes "detuning"; phase unwrap uses that dim as a core dimension on I + jQ

None of these accept a configurable sweep-axis alias for this node path. Renaming only in 03a is therefore sufficient to break the pipeline.

Observed failure

First crash site and error shape

analyse_data → process_raw_dataset → add_amplitude_and_phase(ds, "detuning", subtract_slope_flag=True). Inside add_amplitude_and_phase, _apply_angle calls xarray.apply_ufunc with input_core_dims=[["detuning"]] while I + 1j*Q only has drive_freq_offset_hz on that axis. 

ValueError: tuple.index(x): x not in tuple
  … xarray … apply_ufunc … broadcast_compat_data
  qualibration_libs … add_amplitude_and_phase → _apply_angle
  calibration_utils … process_raw_dataset (line with add_amplitude_and_phase)

Downstream not reached

  • plot_data, update_state, save_results — skipped in that run due to uncaught exception (same Qualibrate action ordering as prior experiments).
Recommendations

How to evolve without surprise

  • Document the contract in the node docstring or README: sweep axis for this frequency sweep must remain detuning unless analysis and plotting are updated together.
  • Parameterize once — e.g. SWEEP_DIM = "detuning" in a shared module used by 03a, analysis.py, and plotting.py (still one name, but a single edit point).
  • Rename for real — rename in all three places and pass the new dim into add_amplitude_and_phase; add a small unit test on a fake xr.Dataset dims tuple.
  • Guardrail (optional) — assert "detuning" in ds.dims (or compare to a constant) at the top of process_raw_dataset with a clear ValueError message listing expected producers.

Related prior test

An earlier experiment broke XarrayDataFetcher multiplex grouping by changing IQ stream tags (I1/Q1 → non-matching names). That failed at convert_IQ_to_V with KeyError: 'I'. Here the IQ tags stayed canonical; only the sweep dimension name changed — different layer, same class of problem: implicit string coupling.
