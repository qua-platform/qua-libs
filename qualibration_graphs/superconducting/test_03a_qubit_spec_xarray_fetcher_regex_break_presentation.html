Stream result tags, XarrayDataFetcher, and downstream IQ analysis

This deck records what was observed when 03a_qubit_spectroscopy.py was changed so that saved stream names no longer match the grouping regex inside qualibration_libs.data.fetcher.XarrayDataFetcher. Execution on hardware still completes; Python analysis then fails with a clear KeyError on I / Q.

Observation

Tags like I1, Q1 are not arbitrary labels: they are the coupling surface between QM job handles and xarray layout.

Hypothesis → outcome

Breaking the regex breaks automatic stacking into variables named I and Q with a qubit dimension.

Evidence

A full run after the tag change: job OK, analyse_data raised KeyError: 'I' inside convert_IQ_to_V.

What ties to what

End-to-end pipeline (contract chain)

Each arrow is a named contract: stream keys from QUA, axis dict keys, xarray variable names, and analysis entry points must stay aligned unless you insert an adapter. 

QUA program (create_qua_program)
  stream_processing():  .save("…")  ──►  QM job result_handles (string keys)

execute_qua_program
  XarrayDataFetcher(job, sweep_axes)  ──►  uses keys + axes to build xr.Dataset

analyse_data
  process_raw_dataset(ds_raw, node)
    convert_IQ_to_V(ds, qubits)     expects  data_vars  "I" and "Q"  (default IQ_list)
    add_amplitude_and_phase(ds, …)  expects  I, Q  after conversion

fit_raw_data
  uses  ds.I, ds.Q, ds.IQ_abs, …   on dimensions  qubit × detuning

Critical link

The fetcher’s regex is the only place that turns I1, I2, … into one variable I stacked on qubit. Rename streams without updating this chain → downstream code never sees I/Q.

Regex mechanics

Why I1 groups and demod_I_slot_1_v does not

The fetcher groups keys only when the entire string matches: a prefix (the “base” name) plus digits only at the end. 

re.match(r"([a-zA-Z_]+?(?:_(c|t)+)?)(\d+)$", key)
Example keyRegex match?Effect in fetcher
I1, Q2YesGrouped into base I / Q; stacked along first axis (qubit).
nNo (no trailing \d+)Unmatched branch: assigned with dims detuning only (already observed in working runs).
demod_I_slot_1_vNoEach key becomes its own data variable; no merge into I.

Intentional experiment (this run only)

For this test, create_qua_program temporarily saved averages as demod_I_slot_{i+1}_v and demod_Q_slot_{i+1}_v so the pattern above did not match — the grouping regex did not fire for those keys. The calibration script was restored to standard I{i+1} / Q{i+1} tags afterward for further work.

Types & shapes

What the dataset looks like before vs after

AspectClassic tags (I1, Q1)New tags (demod_*_v)
Coordinates qubit, detuning (from sweep_axes) Same coordinates registered; unused for unmatched IQ keys on the qubit axis.
Raw IQ variables I(qubit, detuning), Q(qubit, detuning) — numpy arrays inside xarray.DataArray demod_I_slot_1_v(detuning), demod_Q_slot_1_v(detuning) — still real arrays, no qubit dimension on those vars.
Types xarray.Dataset of float-like time-integrated demodulation results (counts before convert_IQ_to_V); job also exposes n for shot progress.

Why analysis fails (precise failure)

  • convert_IQ_to_V does a dict comprehension over ("I", "Q") and indexes the dataset.
  • After the tag change, available keys were reported as demod_I_slot_1_v, demod_Q_slot_1_v only (plus coords) — so KeyError: 'I'.
  • No exception in QUA or on the cluster for this mismatch; it is a pure Python / xarray contract break.
Resilience map

What crashes, what does not, and what is skipped

StageOutcome with broken IQ tags
QUA compile / queue / job runSucceeds — stream names are opaque to the compiler as long as they are unique and consistent with save.
XarrayDataFetcher iterationSucceeds — it still builds a dataset; shape rules for non-matching keys use dims_order[1:] only.
execute_qua_program exitLeaves node.results["ds_raw"] populated with the “wrong” variable names.
analyse_dataRaises (unhandled) — first hard stop at convert_IQ_to_V.
plot_data, update_state, save_resultsNot reached in the same process — Qualibrate run actions short-circuit on uncaught exceptions (so no plots, no QUAM updates from this node in that run).

How to avoid breakage (options)

  • Revert stream tags to I{i+1} / Q{i+1} for this node family.
  • Adapter after fetch: rename / xr.concat demod variables into I and Q with a qubit dimension before process_raw_dataset.
  • Upstream change: extend convert_IQ_to_V / analysis to accept configurable names (larger surface area).

Bottom line: the system does not “crush” silently — hardware and fetching succeed — but the calibration node’s Python tail fails fast with a typed dataset key error until names or adapters are restored.
